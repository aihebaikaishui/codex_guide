# Project Brief

## 1. 项目名称

- 名称：Codex 使用指南

## 2. 项目一句话目标

整理一套适合个人或小团队使用的 Codex 协作方法，让项目在每次任务前都有清晰的目标、边界和状态记录。

## 3. 使用者

- 主要使用者：使用 Codex 协助编写或维护项目的人
- 使用场景：项目初始化、日常迭代、文档维护、任务拆解
- 使用频率：每次让 Codex 执行任务时

## 4. 项目类型

- 文档型开源项目
- 是否需要 Web UI：否
- 是否需要命令行入口：否
- 是否需要数据库：否
- 是否需要调用外部模型 API：否

## 5. 核心功能

### MVP 必须包含

1. Codex 项目四件套的说明
2. 可直接复用的模板
3. 清晰的仓库导航和维护约定

### 后续可以加入

1. 更多任务模板示例
2. 常见问题 FAQ
3. 不同项目类型的适配版本

### 明确暂不做

1. 运行时工具链
2. 在线服务
3. 复杂自动化系统

## 6. 核心流程

1. 使用者阅读主指南。
2. 根据需求补全 `project_brief`。
3. 用 `project_status` 记录当前进度。
4. 每次分配任务时写清楚目标、范围和验收标准。
5. 任务完成后更新状态文档。

## 7. 技术约束

- Python 版本：不适用
- 包管理方式：不适用
- 主要框架：不适用
- 模型调用方式：不适用
- 数据存储方式：Markdown 文档
- 测试框架：不适用
- 不希望引入的依赖：不必要的运行时依赖

## 8. 安全与成本约束

- 不上传敏感文件
- 不在日志中记录密钥或个人隐私
- 不引入必须长期维护的重型工具链
- 不做不可逆的仓库操作

## 9. 验收标准

项目当前阶段完成的判断标准：

1. README 能说明仓库用途和结构
2. 关键开源文件齐全
3. 两份核心模板文档可直接使用

## 10. 当前优先级

按优先级排序：

1. 保持文档清晰可读
2. 保持模板可复用
3. 保持结构足够轻量

## 11. 备注

这是一个偏方法论和模板的仓库，后续如果加入示例项目，再根据实际内容扩展说明。
