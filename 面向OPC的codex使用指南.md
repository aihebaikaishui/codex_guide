# Codex 项目四件套实操配置指南

适用场景：

* 从零创建新的本地 Python AI 项目
* 迭代已有的本地 Python AI 项目

核心目标：

> 用最少的固定配置，让 Codex 知道项目要做什么、该怎么安全地做、当前做到哪、每次任务怎么执行。

推荐四件套：

```text
/docs/project_brief.md      # 项目需求基线文档：定义做什么
AGENTS.md                   # Codex 行为规范：定义怎么做
/docs/project_status.md     # 项目状态文档：记录做到哪
任务指令模板                 # 每次任务的下单格式
```

---

## 一、四件事的关系

### 1. `/docs/project_brief.md`

解决问题：

```text
这个项目到底要做什么？
```

它是 Codex 的需求基线，记录项目目标、核心功能、边界、优先级和验收标准。

### 2. `AGENTS.md`

解决问题：

```text
Codex 在这个项目里应该如何工作？
```

它是 Codex 的行为规则，规定安全边界、确认规则、token 节省规则和文档维护规则。

### 3. `/docs/project_status.md`

解决问题：

```text
项目现在做到哪了？
```

它是项目的长期记忆，记录已完成内容、遗留问题、下一步计划和每次 Codex 任务结果。

### 4. 每次任务使用固定模板

解决问题：

```text
这一次具体让 Codex 做什么？
```

它是单次任务的执行指令，约束本次目标、允许修改范围、测试要求和输出格式。

---

## 二、推荐执行顺序

### 从零创建新项目

推荐顺序：

```text
1. 创建 /docs/project_brief.md
2. 创建 AGENTS.md
3. 创建 /docs/project_status.md
4. 使用固定任务模板让 Codex 开始第一个小任务
```

原因：

* 新项目还没有代码，先写需求基线，避免 Codex 自由发挥
* 再写 `AGENTS.md`，约束 Codex 的工作方式
* 再写 `project_status.md`，建立项目长期记录
* 最后用任务模板逐步开发

---

### 迭代已有项目

推荐顺序：

```text
1. 创建 AGENTS.md
2. 让 Codex 只读分析已有项目
3. 生成或补全 /docs/project_brief.md
4. 生成或补全 /docs/project_status.md
5. 使用固定任务模板继续迭代
```

原因：

* 已有项目存在误改风险，所以先创建 `AGENTS.md` 约束 Codex
* 再让 Codex 只读分析项目，避免一开始就修改代码
* 然后反向整理需求基线和项目状态
* 最后进入正常迭代流程

# 第一件事：创建 `/docs/project_brief.md`

## 为什么要做

`project_brief.md` 是项目需求基线。

没有它，Codex 只能根据当前任务和局部代码猜项目目标，容易出现：

* 功能方向跑偏
* 过度设计
* 引入不必要模块
* 修复 bug 时破坏原始需求
* 多轮任务之间标准不一致

有了它，Codex 每次执行任务前都可以对齐：

* 项目目标
* MVP 范围
* 不做什么
* 核心用户场景
* 技术约束
* 优先级
* 验收标准

---

## 文件位置

```text
/docs/project_brief.md
```

推荐使用这个名字，而不是 `requirements.md` 或 `prd.md`。

原因：

* `requirements.md` 容易和 Python 依赖文件 `requirements.txt` 混淆
* `prd.md` 偏产品经理文档，对个人开发者偏重
* `project_brief.md` 更轻量，适合个人项目和 Codex 使用

---

## 迭代已有项目时怎么生成

已有项目不要一上来让 Codex 全仓乱扫。

推荐指令：

```text
请基于当前已有项目，帮我生成 /docs/project_brief.md。

要求：
1. 先阅读 AGENTS.md
2. 只读分析，不要修改业务代码
3. 优先阅读 README、pyproject.toml、uv.lock、旧依赖文件、src/、tests/ 中必要文件
4. 不做无意义全仓扫描
5. 根据已有代码反向总结项目目标、核心功能、技术约束和当前优先级
6. 如果信息不确定，请标记为“待确认”
7. 创建或更新 /docs/project_brief.md
8. 不要编造不存在的功能
```

# 第二件事：创建 `AGENTS.md`

## 为什么要做

`AGENTS.md` 是 Codex 的项目级行为规范。

它告诉 Codex：

* 哪些操作可以自动执行
* 哪些操作需要先说明
* 哪些操作必须人工确认
* 如何节省 token
* 如何维护文档
* 如何运行测试
* 如何避免危险操作

一句话：

> `project_brief.md` 管目标，`AGENTS.md` 管行为。

---

## 文件位置

```text
AGENTS.md
```

放在项目根目录。

---

## 推荐模板

```markdown
# AGENTS.md

本项目是一个本地 Python AI 项目。Codex 在本仓库中工作时，必须遵守本文件规则。

---

## 1. 必读文件

执行任务前，根据任务需要优先阅读：

- /docs/project_brief.md
- /docs/project_status.md
- 与本次任务直接相关的源码和测试

不要无意义全仓扫描。

---

## 2. 工作原则

- 只完成用户指定任务，不做无关扩展
- 优先阅读必要文件
- 修改前先理解相关上下文
- 修改后运行相关测试
- Python 项目统一使用 `uv` 管理 Python、虚拟环境和依赖
- 依赖操作使用 `uv add`、`uv remove` 或 `uv sync`
- 程序、工具和测试通过 `uv run` 执行
- 不直接使用 `pip`、`Poetry` 或 `Conda`
- 输出保持简洁
- 不输出长篇推理
- 不重复解释用户已经确认过的内容

---

## 3. 编码行为准则

本节借鉴 `andrej-karpathy-skills` 中关于减少 LLM 编码常见错误的原则，并按本项目的 Codex 使用场景改写。

- 不要默默假设需求；如果存在多种理解，先说明差异并确认
- 如果发现更简单的实现路径，应主动说明并优先采用
- 不做用户未要求的功能、抽象、配置项或兼容层
- 不重构与当前任务无关的代码、注释或格式
- 修改应尽量小而集中，每一处改动都能对应到本次任务目标
- 如果本次修改导致 import、变量、函数等变成无用代码，可以清理
- 不删除任务开始前已经存在的无关死代码，除非用户明确要求
- 对非简单任务，先给出简短计划和可验证的成功标准
- 优先通过测试、运行命令或明确检查项验证任务完成

---

## 4. 安全底线

未经用户确认，禁止：

- 删除核心代码文件
- 覆盖配置文件
- 修改生产环境配置
- 数据库结构变更
- 批量删除文件或数据
- 执行高风险系统命令
- 大规模重构
- 修改认证、权限、密钥、支付相关逻辑

---

## 5. 可自动执行的操作

以下操作可直接执行：

- 新增普通业务代码
- 修改非核心逻辑
- 新增或修改测试
- 更新普通文档
- 代码格式化
- 小范围重构
- 修复明确 bug
- 增加类型标注
- 改善错误提示或日志

---

## 6. 需要先说明但可继续执行的操作

以下操作需要先说明影响范围和修改方案；如不触及安全底线，可继续执行：

- 修改多个模块
- 调整核心流程
- 引入新依赖
- 修改项目结构
- 改动数据库访问逻辑
- 修改 AI pipeline 关键步骤
- 修改模型调用逻辑
- 修改 prompt 组织方式

---

## 7. 必须人工确认的操作

遇到以下情况必须停止并等待确认：

- 删除核心代码文件
- 覆盖配置文件
- 修改生产环境配置
- 数据库结构变更
- 批量删除文件或数据
- 执行高风险系统命令
- 大规模重构
- 不可逆操作

---

## 8. Token 节省规则

- 优先阅读必要文件，不做无意义全仓扫描
- 优先基于当前任务做增量分析
- 不重复解释已确认内容
- 不粘贴完整代码，除非用户要求
- 优先引用文件路径、函数名和修改点
- 测试结果只输出关键结论和失败原因
- 长期上下文写入 /docs/project_status.md

---

## 9. 文档维护规则

每次完成独立功能、修复、优化或重构后，必须增量更新：

/docs/project_status.md

要求：

- 不全文重写
- 在文件末尾追加本次任务记录
- 包含目标、完成内容、修改文件、实现思路、遗留问题、下一步、关键指令、测试结果、更新时间

开发每一个模块后，必须增量更新：

/docs/develop_log.md

要求：

- 不全文重写
- 在文件末尾追加或在当前任务对应小节下补充
- 记录模块名称、输入、输出、关键实现结论
- 涉及数据处理时，记录数据规模、字段分布、文件数量、chunk 数量、缺失映射、异常样本等关键统计
- 涉及模型调用时，记录模型配置、调用入口、Token 使用、输出格式、失败原因或异常行为
- 涉及测试或验证时，记录关键指令、样本范围、输出产物和结论
- 不记录 API key、密钥、隐私数据或大段原文

当需求、范围、优先级发生变化时，应更新：

/docs/project_brief.md

要求：

- 只做局部更新
- 不删除重要历史约束
- 不编造未经确认的新需求

---

## 10. 测试规则

优先运行与本次修改直接相关的测试。

如果无法运行测试，必须说明：

- 为什么无法运行
- 哪些部分未验证
- 建议用户如何验证

---

## 11. 输出格式

任务结束时只输出：

- 修改文件列表
- 核心变更摘要
- 测试结果
- 遗留问题
- 下一步建议
```

# 第四件事：每次任务使用固定模板

## 为什么要做

固定模板可以减少沟通成本和误操作。

没有固定模板时，容易出现：

* Codex 不知道允许改哪些文件
* Codex 做无关优化
* Codex 每一步都请求确认
* Codex 忘记跑测试
* Codex 忘记更新文档
* 输出过长，浪费 token

固定模板的作用：

* 明确本次任务目标
* 明确修改边界
* 明确安全规则
* 明确测试要求
* 明确文档更新要求
* 明确输出格式

---

# 4.1 通用任务模板

适合大多数任务。

```text
请基于 AGENTS.md、/docs/project_brief.md 和 /docs/project_status.md 执行本次任务。

任务目标：
【填写具体任务】

背景说明：
【说明当前问题、期望结果、相关模块；没有可写“无”】

允许修改：
- 【文件或目录 1】
- 【文件或目录 2】

禁止操作：
- 不要删除核心代码文件
- 不要覆盖配置文件
- 不要修改生产环境配置
- 不要执行高风险系统命令
- 不要做无关重构

执行要求：
1. 只阅读完成任务所需的必要文件
2. 先简要说明修改方案
3. 低风险修改可直接执行
4. 中等风险修改先说明影响范围；不触及安全底线可继续执行
5. 高风险操作必须等待确认
6. 修改后运行相关测试
7. 完成后增量更新 /docs/project_status.md
8. 如果需求范围发生变化，再局部更新 /docs/project_brief.md

Token 要求：
1. 不做无意义全仓扫描
2. 不输出长篇推理
3. 不粘贴完整代码，除非我要求
4. 优先说明文件路径、函数名和修改点

最终输出：
- 修改文件列表
- 核心变更摘要
- 测试结果
- 遗留问题
- 下一步建议
```

# 七、最小配置总结

最小四件套：

```text
1. /docs/project_brief.md
2. AGENTS.md
3. /docs/project_status.md
4. 固定任务模板
```

最重要的关系：

```text
project_brief.md  = 做什么
AGENTS.md         = 怎么做、哪些不能做
project_status.md = 做到哪、下一步是什么
任务模板           = 这一次具体做什么
```

最推荐的日常使用方式：

```text
1. 让 Codex 先读 AGENTS.md
2. 再读 project_brief.md 确认目标
3. 再读 project_status.md 确认当前状态
4. 最后用固定模板下达单个小任务
5. 任务完成后更新 project_status.md
6. 需求变了才更新 project_brief.md
```

一句话原则：

> 需求写进 `project_brief.md`，规则写进 `AGENTS.md`，进度写进 `project_status.md`，每次任务用固定模板小步推进。
